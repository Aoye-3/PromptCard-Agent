# Local Session, CSRF, And Internal Authentication

The Python Gateway uses a process-local browser session:

1. `POST /agent-api/promptcard/runtime/bootstrap` sets an HttpOnly `access_token` cookie.
2. The Gateway also uses a readable `csrf_token` cookie for double-submit CSRF validation.
3. The frontend sends cookies with `credentials: "include"` and copies `csrf_token` to `X-CSRF-Token` on mutations.

There is no separate user database, JWT account lifecycle, setup wizard, or visible login form.

Local service-to-service calls use `X-PromptCard-Internal-Token`:

- Python Gateway to pi `/invoke`
- pi to Python Gateway `/internal/chat`

The internal token is generated by the combined launcher when absent. It is not exposed to browser code and does not replace CSRF protection for browser requests.
